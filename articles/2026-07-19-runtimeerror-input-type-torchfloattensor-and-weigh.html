How to Fix the "RuntimeError: Input type and weight type should be the same" in PyTorch

Source: Stack Overflow ↗ · 201 votes

I still remember the first time I saw this error back when I was training my first custom CNN. I had spent nearly four hours debugging my data pipeline, convinced that my dataset was corrupted or that my GPU drivers were failing, only to realize I had simply forgotten to move my tensors over to the CUDA device. I felt like a total amateur staring at the screen, but it’s honestly one of those "rite of passage" errors that every PyTorch developer runs into at least once.

The root cause of this headache is actually pretty simple: PyTorch is incredibly strict about hardware parity. When you define your model, you move it to the GPU using model.to(device), which turns its internal weights into cuda.FloatTensor types. However, when you iterate through your DataLoader, those input tensors are created on your CPU by default as torch.FloatTensor. When you pass those CPU-bound inputs into a GPU-bound model, PyTorch throws a fit because it can’t perform mathematical operations across different memory spaces. It’s basically trying to plug a European power adapter into an American wall outlet—the hardware just doesn't speak the same language.

To fix this, you need to manually move your input tensors onto the same device as your model during your training loop. I usually handle this right after I unpack my data from the loader. Here is the snippet that saves the day:


device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model.to(device)

for data in dataloader:
    inputs, labels = data
    # Move the inputs and labels to the device explicitly
    inputs = inputs.to(device)
    labels = labels.to(device)
    
    outputs = model(inputs)
    # ... rest of your training logic

Once you add those .to(device) calls, the tensors are copied to your GPU memory before the model ever sees them, and the error vanishes instantly.

If you want to prevent this from ruining your afternoon in the future, my best advice is to write a small helper function or a custom Dataset class that handles device placement automatically, or just get into the habit of writing your training loop with those lines included by default. I’ve started using a simple batch_to_device function in my boilerplate code, so I don't have to think about it every time I start a new project.

Happy coding, and hopefully, you won't have to spend nearly as much time staring at this error as I did!

PyTorch 報錯 RuntimeError：當你的模型在 GPU，資料卻還在 CPU 的崩潰日常

來源：Stack Overflow ↗ · 201 票

還記得剛開始接觸 PyTorch 的時候，我整個人充滿雄心壯志，想著終於要把那個跑了半天的模型丟上 GPU 加速了。結果程式一跑下去，螢幕瞬間跳出一大串紅字 RuntimeError: Input type (torch.FloatTensor) and weight type (torch.cuda.FloatTensor) should be the same，當時我盯著螢幕看了十分鐘，內心真的快崩潰，心想：「我不是都寫了 .to(device) 了嗎？為什麼還是不理我！」

其實這個錯誤的原因很單純，就是你的模型雖然乖乖搬到 GPU 上了，但你的輸入資料（input）還懶洋洋地留在 CPU 的記憶體裡。PyTorch 非常嚴格，它不允許模型在顯卡上運算時，吃進去的資料卻還在系統記憶體，這就好比你把廚師請進了高級餐廳的廚房，卻沒把食材送進去，廚師當然會罷工給你看。那個當下最痛苦的，通常是明明感覺自己邏輯都對了，卻被這種「資料位置不同步」的小細節卡死，還要花時間去 Debug，真的會讓人懷疑人生。

解決的方法其實很直觀，你只需要在迴圈讀取資料的時候，順手把資料也推到 GPU 上就好。我通常會在 dataloader 的迴圈裡，直接對 inputs 和 labels 做處理。你可以直接這樣寫：


device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model.to(device)

for data in dataloader:
    # 這裡就是關鍵！把資料也丟過去
    inputs, labels = data
    inputs = inputs.to(device)
    labels = labels.to(device)
    
    outputs = model(inputs)
    # 接下來的 loss 計算也要記得 labels 已經在 GPU 上了

這樣一改，程式就能順利跑起來了。為了避免以後再犯這種低級錯誤，我現在習慣養成一個反射動作：只要定義了 device 變數，後續任何跟張量（Tensor）有關的初始化，或是從 DataLoader 拿出來的資料，我都會強制性地加上 .to(device)。甚至後來我會寫一個小工具函式，專門用來把整批資料轉移，這樣代碼看起來比較乾淨，也不容易漏掉。

搞定這個小坑之後，GPU 的運算效率應該就能讓你很有感了，祝大家跑模型都能一次過，別再被這種錯誤卡住啦！
