Debugging the "Hydration Failed" Error in Next.js and React 18

Source: Stack Overflow ↗ · 341 votes

I still remember the first time I saw the "Hydration failed" error in a Next.js project. It was 2:00 AM, I was three cups of coffee deep, and I had spent nearly four hours trying to figure out why my production build looked like a flickering mess of broken HTML. I kept checking my API routes and database queries, convinced it was a backend issue, only to realize I had blindly dropped a new Date() call directly into my JSX.

The root cause of this nightmare is actually quite simple: a mismatch between what the server renders and what the client expects. When Next.js pre-renders your page on the server, it generates a static snapshot of the HTML. When that HTML hits the browser, React attempts to "hydrate" it by attaching event listeners and turning it into a dynamic app. If the server says "it's 10:00 AM" but your browser—running on a different timezone or simply a millisecond later—says "it's 10:00:01 AM," React panics. It sees the discrepancy, assumes the entire DOM tree is corrupted, and throws that dreaded hydration error. It’s essentially React’s way of saying, "Wait, this isn't what I signed up for," and then it proceeds to tear down the server-rendered UI to replace it with a client-side version.

I fixed this by ensuring that the initial render remains identical on both the server and the client. If you have content that is inherently dynamic—like timestamps, random numbers, or local storage data—you have to tell React to wait until the component has officially mounted on the client before rendering that specific piece of UI. I usually handle this by using a simple useEffect hook to toggle a state variable once the component is ready.


import { useState, useEffect } from "react";

export default function DynamicComponent() {
  const [isClient, setIsClient] = useState(false);

  useEffect(() => {
    setIsClient(true);
  }, []);

  return (
    <div>
      <h1>Welcome to my app</h1>
      {isClient ? (
        <p>Current Time: {new Date().toLocaleTimeString()}</p>
      ) : (
        <p>Loading...</p>
      )}
    </div>
  );
}

By adding that isClient check, I ensure that the server renders the "Loading..." state, and the browser handles the actual date rendering only after the hydration process is complete. This keeps the DOM tree consistent during that critical first paint.

To prevent this from happening again, I’ve made it a rule to always audit my components for non-deterministic data. If you’re using third-party libraries that rely on the window object or checking local storage, wrap them in a custom hook or a conditional check that defaults to a static, server-friendly value first. It saves you a massive headache down the road and keeps your app’s performance smooth.

Hopefully, this saves you from losing as much sleep as I did back then. Catch you later, and happy coding!

React 18 的 Hydration 錯誤：別讓前後端的不一致毀了你的週末

來源：Stack Overflow ↗ · 341 票

記得有次半夜趕專案，正當我自信滿滿地準備把 Next.js 專案推上 Production 時，瀏覽器控制台突然噴出一大堆紅字，寫著「Hydration failed」。那一刻我真的心跳漏了一拍，明明開發環境跑得好好的，怎麼一上線就原地爆炸？最讓人崩潰的是，網頁看起來雖然沒壞，但卻因為這個錯誤導致部分互動功能失效，那種「明明程式碼沒動，卻在最後一刻被背刺」的無力感，相信大家一定都懂。

其實這個問題的核心在於「前後端認知不一致」。簡單來說，Next.js 在伺服器端（SSR）會先渲染出一段 HTML 傳給瀏覽器，但當 React 在瀏覽器端（Client）接手並執行 Hydration 時，它發現自己生成的 DOM 結構跟伺服器給的不一樣。這通常是因為我們在渲染時用了某些會隨機變動的值，例如 Math.random()、當前的時間，或是從 localStorage 讀取的資料。伺服器渲染時這些值是一個樣，瀏覽器執行時又變成另一個樣，React 一看：「欸，這根本不是我預期的那個樣子！」於是就直接報錯投降了。

解決這個問題的邏輯其實很直觀：我們要確保這段會變動的程式碼，只在瀏覽器端執行，不要讓伺服器去插手。最常見的做法是利用 useEffect，因為它只會在元件掛載到瀏覽器後才執行。我通常會這樣寫：先宣告一個 state 來判斷是否已經在瀏覽器端渲染完成，如果還沒，就先顯示一個簡單的 Loading 狀態或是空標籤。


import { useState, useEffect } from 'react';

function MyComponent() {
  const [isClient, setIsClient] = useState(false);

  useEffect(() => {
    setIsClient(true);
  }, []);

  return (
    <div>
      {isClient ? (
        <p>現在時間是: {new Date().toLocaleTimeString()}</p>
      ) : (
        <p>Loading...</p>
      )}
    </div>
  );
}

透過這種方式，伺服器渲染時會統一輸出「Loading...」，瀏覽器接手後再切換成真實的時間，這樣前後端就達成共識了，錯誤自然就消失。

至於要怎麼預防下次再踩到這種雷？我的建議是養成習慣，只要涉及到瀏覽器特有 API（像是 window、localStorage 或是不穩定的隨機值），一律包在 useEffect 裡處理。如果在寫元件時心裡閃過「這東西在伺服器上有意義嗎？」的念頭，那就直接把它丟進 useEffect 就對了，省得之後還要回來修這些莫名其妙的 Hydration 錯誤。

除錯這條路就是這樣，坑踩多了，下次看到紅字就不會想把電腦丟出去了。希望這招能幫你避開那個該死的錯誤，祝大家 Coding 順利！
