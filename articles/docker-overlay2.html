Docker Linux 2026-07-15

docker system prune 跑完，硬碟還是滿的？我是怎麼找到那個吃掉 40GB 的元兇

執行了 docker system prune -a --volumes 卻發現 /var/lib/docker/overlay2 依然肥得像座山？本文帶你用一段腳本精準抓出元兇容器，並用一行指令清空它的 Log，服務完全不需要重啟。

說起來有點丟臉，但我覺得這個坑應該很多人踩過。

上週四深夜，我的一台 staging server 突然就炸了。CI/CD pipeline 開始噴各種奇怪的錯誤，一開始還以為是程式的 bug，仔細一查，才發現根本原因就一行字：

No space left on device

好，我見過這個。深吸一口氣，打開終端機，輸入那行我已經背得滾瓜爛熟的指令：

docker system prune -a --volumes

系統開始跑，清了差不多 8GB，我心想「好，搞定了」。

結果 df -h 一看，根目錄還是 97% 滿。

問題出在哪？（我花了兩個小時才搞清楚）

docker system prune 這個指令其實只會清理「孤兒」——就是那些沒有被任何容器使用的映像檔、網路、和掛載卷。它壓根不會動正在跑的容器裡面的東西。

而我的問題，就出在一個跑了快三個月、一直沒重啟的容器上。

你猜那個容器裡面是什麼？是 Log。

Docker 預設的 Log 機制有個很坑的設計：它不會自動限制 Log 檔案的大小。你的應用程式每印一行紀錄，它就忠實地幫你記下來，永無止境，直到把硬碟塞滿為止。我那個容器的 log 檔長到了 43GB。

怎麼找到那個「容量怪獸」

overlay2 裡面的資料夾名稱都是一串看不懂的 Hash，你根本不知道是哪個容器對應到哪個資料夾。所以你得用這段指令，把 Hash 翻譯回人話：

docker ps -q | xargs docker inspect \
  --format '{{.Name}} {{.GraphDriver.Data.WorkDir}}' | \
while read name workdir; do
  size=$(sudo du -sh "${workdir%/work}" 2>/dev/null | cut -f1)
  echo "$size  $name"
done | sort -h

跑完之後，你會看到一個清單，大小從小排到大。最後幾行就是你要找的目標。

實際解決：一行指令清空 Log，容器照常運行

找到那個可惡的容器之後，先用這個指令拿到它的 ID：

docker inspect --format '{{.Id}}' <你的容器名稱>

然後用 truncate 清空 Log 檔。這個指令的重點是，它不會停掉你的容器，服務完全不受影響：

truncate -s 0 /var/lib/docker/containers/<container_id>/*-json.log

我跑完這行，立刻從 97% 掉回 32%。

那種感覺真的很爽，就像是幫伺服器做了一次大掃除。

最重要的部分：以後不要再讓這件事發生

解決問題只是治標。真正的解法是，在你的 docker-compose.yml 裡面幫每個服務加上 Log 限制。

官方文件其實有提這個，但藏得很深，而且沒有強調「你不設的話日後一定會後悔」。所以很多人都不知道這件事：

services:
  my_app:
    image: my_app:latest
    logging:
      driver: "json-file"
      options:
        max-size: "50m"
        max-file: "3"

這樣設定之後，這個服務的 Log 最多就是 150MB（50MB × 3 個檔輪替），再也不會無限長大了。

改完記得重新 docker-compose up -d 讓設定生效。
