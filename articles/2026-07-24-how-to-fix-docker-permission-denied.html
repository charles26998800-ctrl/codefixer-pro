Fixing the "Docker: Permission Denied" Error on Ubuntu

Source: Stack Overflow ↗ · 1416 votes

I still remember the first time I set up Docker on a fresh Ubuntu install, feeling like a total pro until I ran my first container. I spent a good hour convinced I’d somehow corrupted my entire system installation because every command I typed ended with that dreaded "permission denied" wall. It’s honestly one of those classic "welcome to Linux" moments that makes you want to throw your keyboard out the window when you’re just trying to get a simple project off the ground.

The reason you're hitting this wall is actually pretty straightforward once you peel back the layers. By default, the Docker daemon binds to a Unix socket owned by the root user, not your local user account. When you run a command without sudo, the Docker client tries to communicate with that socket, but the Linux kernel shuts it down immediately because your user doesn't have the required group privileges. It’s a security feature designed to keep unauthorized users from poking around the daemon, but it’s definitely an annoyance when you’re just trying to run your local development environment.

To get past this without typing sudo every single time, you need to add your user to the docker group. I usually start by creating the group if it doesn't already exist, though the Docker installation usually handles this part for you. From there, I add my current user to that group so I can interact with the socket directly. Here is the sequence of commands I run to fix it:


# Add the docker group if it doesn't exist
sudo groupadd docker

# Add your current user to the docker group
sudo usermod -aG docker $USER

Once you’ve run those, the changes won't take effect immediately. You’ll need to either log out and log back in, or run newgrp docker to refresh your terminal session’s group membership. After that, you can run docker run hello-world without the sudo prefix and everything should fire up perfectly.

If you want to avoid dealing with this in the future, my best advice is to handle your user permissions immediately after installing the Docker engine. I always treat adding my user to the docker group as a mandatory step in my "new machine setup" script, right alongside installing git and setting up my SSH keys. It saves me from the headache of typing sudo for every single command and makes the development workflow feel significantly more seamless.

Hope this clears up the confusion and gets you back to coding. Let me know if you run into any other weird permission hiccups!

為什麼 Docker 老是嫌我沒權限？教你一招搞定 Permission denied

來源：Stack Overflow ↗ · 1416 票

剛開始接觸 Docker 的時候，我真的差點崩潰。那時候我正趕著把一個小專案部署到 Ubuntu Server 上，心想一切都安裝好了，結果輸入 docker run hello-world 卻跳出一串紅通通的錯誤訊息，說我沒有權限連接到 Docker daemon。我當時心裡的第一個反應是：「蛤？我都已經是 root 權限了還想怎樣？」最讓我痛苦的時刻，就是每次執行指令都要加上 sudo，搞得我手指都快抽筋，而且在那之後的一整天，我都在跟這些權限設定奮戰，感覺像是在跟作業系統玩躲貓貓。

其實這問題的根本原因很單純，Docker daemon 預設是綁定在 Unix socket 上的，而這個 socket 檔案的擁有者是 root。當你沒有用 sudo 時，你的使用者帳號根本沒有權限去觸碰那個檔案，所以 Docker 才會傲嬌地拒絕你的請求。簡單來說，就是你的帳號沒有被加入到 Docker 的「特權俱樂部」裡。

要解決這個問題，我們不需要每次都靠 sudo 活著。你只需要把你當前的使用者帳號加入到 docker 這個群組裡就好。我當時是這麼處理的，先建立一個 docker 群組（雖然通常安裝完就有了），接著把我的使用者拉進去：


# 建立 docker 群組（如果系統提示已存在就不用理它）
sudo groupadd docker

# 把目前的使用者加入 docker 群組
sudo usermod -aG docker $USER

設定完之後，重點來了，你必須要「登出再重新登入」，或者直接執行 newgrp docker，這樣系統才會重新整理你的群組權限。如果你懶得登出，newgrp 是救星，它能讓你不用重開機就直接生效。

為了預防下次再被這種權限問題搞到心累，我現在養成的習慣是，安裝完 Docker 後第一件事就是把這段指令寫進我的安裝腳本裡。只要確認你的使用者在 docker 群組裡，以後開發時根本不需要再多打這五個字母，體驗真的會好上非常多。

以後看到那串錯誤訊息，別再傻傻地一直加 sudo 了，直接把權限處理好，開發效率才會高啊！
