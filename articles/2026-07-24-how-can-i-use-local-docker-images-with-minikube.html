Stop Pushing to Docker Hub: How to Use Local Images with Minikube

Source: Stack Overflow ↗ · 688 votes

I still remember the first time I set up a local Kubernetes cluster. I spent three hours pulling my hair out because I kept changing a single line of code in my Go service, rebuilding the image, and watching Minikube fail to pull the "new" version because it was still using the cached one from Docker Hub. I was burning through my data plan and my patience, convinced I had to push every tiny iteration to a remote registry just to see if my fix worked.

The reason this happens is actually pretty simple: Minikube runs inside its own isolated virtual machine or container environment. When you run docker build on your host machine, that image lives in your host’s Docker daemon. Minikube, however, is looking for images inside its own internal Docker daemon. It doesn't know your host machine exists, so when it tries to pull an image, it checks its own internal storage, then checks the remote registry, and ignores your local machine entirely.

The fastest way I’ve found to bridge this gap is to point your host's Docker CLI directly at the Minikube daemon. You can do this by running a single command in your terminal before you build your images:


eval $(minikube docker-env)

Once you run that, any docker build command you execute will build the image directly inside the Minikube environment. It’s like magic—the image is suddenly "there" as far as Kubernetes is concerned. Just remember that if you open a new terminal tab, you’ll need to run that command again, as it only affects your current shell session. If you’re using Kubernetes manifests, make sure you set imagePullPolicy: IfNotPresent in your deployment YAML; otherwise, K8s might still try to reach out to a remote registry even if the image is already sitting right there in the local cache.

To save yourself the headache in the future, just get into the habit of checking your environment variables whenever you're doing local development. If you ever feel like you're going crazy because your changes aren't reflecting in your pods, run docker images inside your terminal to see if the image tag you just built actually exists in the Minikube registry.

Hope this saves you from the same three-hour rabbit hole I fell down. Happy coding!

別再把 Image 推上 Docker Hub 了！Minikube 使用本地映像檔的正確姿勢

來源：Stack Overflow ↗ · 688 票

記得剛開始碰 Kubernetes 的時候，我為了測試一個小小的改動，每次都要把 Image 打包、Push 到 Docker Hub，然後再到 Minikube 裡 Pull 下來。那天下午我重複了這個動作不下二十次，看著終端機顯示 Pulling image... 的進度條，我的心態真的快崩了，完全是在浪費生命等待網路傳輸。最慘的是，好不容易傳上去發現程式碼還有個 typo，當下真的只想把筆電闔上直接下班。

其實這背後的原因很單純，因為 Minikube 本質上是一個虛擬機（或是容器），它有自己獨立的 Docker 環境，跟你電腦本機的 Docker Daemon 是隔離的。當你在本地執行 docker build 時，Image 是存在你電腦的 Docker 環境裡，Minikube 的 Node 根本「看不見」這些東西，所以它只能乖乖地去遠端 Registry 拉取。

後來我發現，根本不需要這麼麻煩。最快的方法就是直接告訴你的終端機：「嘿，現在請對著 Minikube 內部的 Docker 環境操作」。你只要執行一行指令，把環境變數導向過去：


eval $(minikube docker-env)

執行完這行之後，你在同一個視窗內執行的 docker build -t my-app:v1 .，就會直接在 Minikube 的虛擬機裡建置，Kubernetes 就能直接抓到它了。記得要把你的 Deployment YAML 裡的 imagePullPolicy 改成 IfNotPresent，不然 Kubernetes 還是會不死心地想去網路上找這個 Tag 的 Image。

如果你懶得每次都設定這個環境變數，或是覺得這樣切來切去很煩，我建議乾脆養成一個習慣：在開發階段就寫一個小小的 Makefile。把 eval $(minikube docker-env) 和你的 build 指令包在一起，甚至寫個簡單的 script 來自動處理。這樣以後只要打 make build，一切就自動搞定，再也不用為了等 Image 拉取而盯著螢幕發呆了。

反正開發嘛，能少按一次鍵盤就少按一次，把時間省下來喝杯咖啡不是更好嗎？
