How to Undo Your Last Local Git Commit Without Losing Your Mind

Source: Stack Overflow ↗ · 27225 votes

I still remember the first time I accidentally committed a massive dump of debug logs and sensitive configuration files to a local branch. I spent nearly two hours sweating over whether I’d accidentally pushed the mess to the company server, only to realize I was just staring at my own local history. We’ve all been there—that sinking feeling when you hit "Enter" on a git commit and immediately realize you forgot to stage a crucial file or included something you definitely shouldn't have.

This usually happens because Git is designed to be a snapshot-based system. When you run git commit, you’re essentially sealing a moment in time into your project's history. It’s a great feature for accountability, but it’s a total headache when you’re moving fast and your fingers get ahead of your brain. Since you haven't pushed your changes to a remote repository yet, Git is perfectly happy to let you rewrite that history. You aren't actually deleting the work; you're just shifting the pointer of your local branch back to a previous state.

To fix this, I always reach for git reset. If you just want to undo the commit but keep your changes in your staging area—maybe to add one more file you forgot—you’ll want to use a soft reset. Just run this in your terminal:


git reset --soft HEAD~1

If you realize you made a complete mess of the code and want to wipe out the commit *and* all the changes associated with it entirely, you should use a hard reset instead. Be careful with this one, because it’s destructive:


git reset --hard HEAD~1

Once you run these commands, your branch pointer moves back one step. If you chose the soft reset, your files will be sitting in your staging area, ready for you to adjust them. If you went with the hard reset, it’s as if those changes never existed in your working directory.

The best way to prevent this headache is to adopt the habit of running git status before every single commit. I started forcing myself to look at that list of green and red files every time I type git commit -m, and it’s saved me from countless "oops" moments. It takes an extra three seconds, but it beats the panic of trying to surgically remove a bad commit from your history later.

Hope this saves you some stress next time you make a commit slip-up. Happy coding!

手殘把垃圾 code commit 上去了？Git 撤銷 commit 的救命指南

來源：Stack Overflow ↗ · 27225 票

記得有一次凌晨三點，我為了趕一個緊急 hotfix，腦袋已經呈現「當機」狀態。當時我隨手 git add 把整個資料夾都塞了進去，按完 commit 才驚覺我把還在實驗階段、甚至會讓程式直接 crash 的垃圾測試檔也一起包進去了。那一刻我真的心臟漏跳一拍，看著那個剛剛產生的 commit ID，心裡想著：「完了，這要是推上去，明天早上主管絕對會找我喝咖啡。」那種冷汗直流的感覺，相信每個工程師應該都懂。

其實這問題發生的根本原因很單純，就是我們太依賴 git add . 這個指令，卻忘了在 commit 前先檢查一下 git status。Git 的設計邏輯是把 commit 當作一個「快照」，一旦你下達了這個指令，Git 就會把暫存區（Staging Area）的所有東西封裝起來。在還沒 push 到遠端之前，這些 commit 其實都還乖乖待在你的本地端，就像是還沒寄出的信件，只要你還沒按「發送」，一切都還有救。

當我發現自己闖禍時，我第一時間就是用 git reset 來處理。如果你只是單純想撤銷最後一次 commit，但想保留修改過的檔案內容，讓我可以在修正後再重新 commit，我通常會用這個指令：


git reset --soft HEAD~1

這行指令的魔力在於，它會把最後一個 commit 拆開，但你會發現你改寫的那些檔案通通都還在「暫存區」裡，就像你剛剛才執行完 git add 一樣。如果你的狀況更慘，是連檔案內容都想直接丟掉、想讓程式碼直接回到上一個乾淨的狀態，那就要改用暴力一點的：


git reset --hard HEAD~1

不過用 --hard 之前千萬要三思，因為它會直接把你這一次 commit 裡的所有修改全部抹除，真的會連渣都不剩，所以我通常只在確定那次 commit 根本是場災難時才會用。

為了避免這種心驚膽跳的情況重演，我後來養成了一個習慣：在 git commit 前，一定會再檢查一次 git status。如果還是怕手抖，我會強烈建議大家改用 git add -p（patch mode），它會一個區塊一個區塊問你：「這個改動要加入嗎？」這樣你就能精準控制到底要 commit 哪些檔案，而不是每次都把一堆亂七八糟的垃圾檔案全塞進去。

學會這幾招，以後再也不用怕手殘 commit 了，現在可以安心去喝杯咖啡啦！
