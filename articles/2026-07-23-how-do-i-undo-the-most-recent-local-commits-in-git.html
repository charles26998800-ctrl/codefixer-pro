How to Undo Your Most Recent Local Git Commits Without Panicking

Source: Stack Overflow ↗ · 27227 votes

I still remember the first time I accidentally committed my .env file containing local API keys to a repository. I spent about two hours sweating bullets, convinced I’d just handed the keys to my database to the entire internet, before a senior dev walked by, laughed at my panicked expression, and showed me the git reset command. It was a humbling lesson, but one that saved my skin, and I’ve used that same trick dozens of times since whenever I’ve mismanaged my staging area.

The reason this happens is that Git is designed to be a ledger of your work, and it treats every commit as a permanent milestone by default. When you run git commit, you're telling Git to take a snapshot of your staged changes and lock them into the project history. If you realize immediately that you've included a file you didn't mean to, or if you simply made a typo in your commit message, you're technically trying to rewrite history that Git thinks is already set in stone. It’s not that you've done something wrong, it's just that Git takes "committing" very seriously.

When I need to undo that last commit, I rely on git reset. The command you’ll want to run depends on whether you want to keep your work or toss it entirely. If you want to undo the commit but keep the files in your staging area so you can fix them and commit again, I use git reset --soft HEAD~1. This keeps all your changes ready to be re-committed, which is perfect for fixing a bad commit message or adding that one forgotten file.

If you made a total mess of things and want to completely wipe the commit and the changes associated with it, I use the --hard flag. Just be careful with this one, because it’s destructive:


git reset --hard HEAD~1

This command rolls back the repository to the state it was in exactly one commit ago, and it permanently deletes the changes you made in that specific commit. If you just want to unstage the files but keep your work in the directory, use git reset HEAD~1 without any extra flags. It’s the safest middle ground, as it removes the commit while leaving your code files untouched and ready for a fresh start.

To prevent this from becoming a recurring headache, I’ve started using git status like a nervous tick. Before I run any commit command, I force myself to look at the list of staged files one last time. If you’re really worried about hitting the wrong key, you can even set up a pre-commit hook that runs a script to check for sensitive files—it’s saved me from a lot of embarrassing public repo mistakes.

Hopefully, this helps you clean up your history without losing your mind. Happy coding, and may your Git history always be clean!

手滑 Commit 了怎麼辦？Git 撤銷本地紀錄的保姆級教學

來源：Stack Overflow ↗ · 27227 票

那天半夜兩點，我正趕著把一個小功能補完，結果一陣昏頭，把測試用的 Debug Log 跟一些亂七八糟的設定檔全都一起 git commit 了，更慘的是，我還在沒檢查的情況下習慣性地按了 Enter。當下看著螢幕上那一長串雜亂的 Commit Message，我心臟真的漏跳一拍，那種「完了，整組壞掉」的恐懼感直接湧上來，差點沒把鍵盤給砸了。

其實這問題說穿了，就是 Git 把你的修改紀錄在本地的「歷史清單」裡，而我們只是想把這個清單往回翻一頁而已。在 Git 的世界裡，Commit 就像是存檔點，如果你還沒把這些東西推送到遠端伺服器（Push），那這些紀錄其實都還在你的電腦裡，隨時都可以修改。最怕的就是那種「已經 Push 上去」才發現錯了，那才真的會被同事白眼到死。

遇到這種情況，我通常會用 git reset 這個指令。如果你只是想把剛剛那次 commit 撤掉，但想保留修改過的檔案內容（方便你繼續改），直接輸入 git reset --soft HEAD~1 就好。這招超級好用，它會把那個 commit 拆掉，但你的程式碼還會乖乖地躺在「暫存區（Staging Area）」，這時候你只要把多餘的檔案從暫存區拿掉再重新 commit 就行。

如果是想徹底清乾淨，連修改的內容都不想要了，那就要用 git reset --hard HEAD~1。這招比較狠，會直接把你最後一次 commit 的所有更動全部抹掉，恢復到還沒改之前的樣子。我第一次用這招時，因為沒確認檔案有沒有備份，結果把辛辛苦苦打了一晚上的邏輯全弄丟，當下盯著螢幕發呆了五分鐘，那種心涼的感覺到現在都還記得。所以啊，用 hard 之前真的要先確認過，不然真的會哭出來。

為了避免這種慘劇再次發生，我現在養成了「先 Git Status 再 Commit」的強迫症。在輸入 commit 訊息之前，我一定會先打 git status 看一下到底有哪些檔案被加進去了，甚至會用 git diff --cached 檢查一下改了什麼。雖然多花這幾秒鐘很煩，但比起半夜在那邊修復 Git 紀錄，這點時間絕對值得。

下次手滑的時候先別慌，深呼吸一下，Git 的回復機制通常都能救你一命的。祝大家 Commit 順利，永遠都不會遇到需要 Undo 的那一刻！
