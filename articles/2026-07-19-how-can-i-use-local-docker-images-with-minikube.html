How to Use Local Docker Images with Minikube Without Pulling Your Hair Out

Source: Stack Overflow ↗ · 688 votes

I still remember the first time I set up a local Kubernetes cluster. I spent three hours banging my head against the wall because my pods kept stuck in ImagePullBackOff, even though I was staring right at the image in my local Docker daemon. I felt like a total amateur, convinced that my YAML was somehow cursed, before I realized that Minikube lives in its own little world and doesn't actually care what's happening on my host machine’s Docker engine.

The reason this happens is simple: Minikube runs inside a virtual machine or a separate containerized environment. When you run docker build on your laptop, that image is stored in your host's Docker socket. When you tell Kubernetes to pull an image, it looks inside its own specific environment, sees nothing, and immediately tries to reach out to a public registry like Docker Hub. Since your custom image isn't there, the deployment fails, and you’re left with a frustrating cycle of pushing and pulling just to test a one-line code change.

The quickest way I’ve found to bridge this gap is to point your local terminal directly at the Minikube Docker daemon. Instead of pushing to a registry, you just run a quick command to tell your shell to use the environment inside Minikube. Run this in your terminal:


eval $(minikube docker-env)

Once you run that, any docker build command you execute will build the image directly inside the Minikube environment. You can verify it worked by running docker images, and you'll see your local images sitting right there where Kubernetes can actually see them. Just make sure your deployment's imagePullPolicy is set to IfNotPresent in your YAML file, otherwise, Kubernetes will still try to go out to the internet to check for a newer version of the image.

To keep this from becoming a recurring headache, the best habit you can pick up is to always tag your images with a specific version or a unique identifier like the commit hash. If you rely on the latest tag, Kubernetes might get confused if you've updated the image locally but didn't explicitly restart the pod, leading to some truly bizarre debugging sessions where the code you're running doesn't match the code you think you're running.

Hope this saves you the hours of frustration I went through when I was starting out. Happy coding, and may your deployments stay green!

別再把 Image 推到 Docker Hub 了！Minikube 使用本地映像檔的正確姿勢

來源：Stack Overflow ↗ · 688 票

記得剛開始碰 Kubernetes 的時候，我為了測試一個小小的改動，硬是把 Docker Image 推到遠端 Registry 再拉下來，光是等網路傳輸就夠我喝兩杯咖啡了。最崩潰的是，好不容易跑起來發現有個 typo，改完又要再走一次這個漫長的流程，那天下午我真的差點對著螢幕翻桌，心想：「難道工程師的時間就是這樣被浪費掉的嗎？」

其實這背後的邏輯很簡單，Minikube 是一個跑在虛擬機（或容器）裡的獨立環境，它就像你電腦裡的一個「孤島」。當你在主機上執行 docker build 時，映像檔是存在你本機的 Docker Daemon 裡，但 Minikube 根本看不見這個島以外的地方。所以當你寫好 YAML 檔要部署時，K8s 找不到那個 Image，就會乖乖跑去網路找，找不到就直接給你一個 ImagePullBackOff 的錯誤，那個紅色大字看了真的會讓人心跳漏半拍。

後來我發現，其實不用那麼麻煩，最快的方法就是直接把環境變數切換過去。只要在終端機輸入這行指令：


eval $(minikube docker-env)

執行完這行後，你的終端機就會暫時「接管」Minikube 內部的 Docker 環境。這時候你再執行 docker build -t my-app:v1 .，這個 Image 就會直接蓋在 Minikube 的肚子裡。之後你在 Deployment 的 YAML 檔裡記得把 imagePullPolicy 改成 IfNotPresent，這樣 K8s 就會優先抓取本地已經有的映像檔，再也不用在那邊苦苦等待上傳下載了。

為了避免以後又因為忘記切換環境而在那邊鬼打牆，我現在養成了習慣，每次開一個新的開發 Session，第一件事就是檢查 minikube status。如果是在做 CI/CD 的測試，我建議乾脆寫一個簡單的 Makefile，把這些環境切換和建置指令都包進去，這樣不僅省力，也不會因為一時手殘忘記切換環境而浪費寶貴的開發時間。

好啦，希望這招能幫你省下不少等網路傳輸的空檔，快去寫扣吧！
