How to Check If a File Exists in Python Without Using Try-Except Blocks

Source: Stack Overflow ↗ · 7349 votes

Early in my career, I remember building a file-processing script that was supposed to handle hundreds of logs every morning. I was so proud of my try-except blocks until I realized I’d accidentally wrapped my entire logic in a single broad exception, causing my script to silently swallow legitimate errors instead of just checking if the input file existed. I spent four hours debugging why my program wasn't failing when a file was missing, only to realize I had essentially created a black hole for my own error messages.

The reason this feels so tricky is that Python’s philosophy is often "Easier to Ask for Forgiveness than Permission" (EAFP). In many cases, using try and except is actually the "Pythonic" way to handle file operations because it avoids race conditions. If you check if a file exists, then try to open it, another process might delete that file in the millisecond between your check and your action. However, when you just need a quick boolean check to decide whether to skip a task or log a warning, throwing an exception feels like overkill.

To solve this without cluttering my code with blocks, I’ve moved over to the pathlib module. It’s built into the standard library and honestly makes file handling feel like actual modern programming. Here is how I usually handle it now:


from pathlib import Path

file_path = Path("my_data.txt")

if file_path.exists():
    print("The file is right where it should be.")
else:
    print("File not found, skipping this step.")

If you’re working with older codebases or just prefer the classic os module, you can still achieve this easily. While pathlib is more object-oriented, the os.path utility is just as functional for simple existence checks.


import os

if os.path.exists("my_data.txt"):
    print("Confirmed: the file exists.")

One concrete tip to save your future self some headaches: always define your paths as variables at the top of your functions or modules rather than hardcoding strings deep inside your logic. When you keep those paths centralized, you can easily swap them out for testing or configuration changes without hunting through your entire codebase, which is exactly how I finally stopped chasing those "file not found" ghosts in my own projects.

Hope this clears up the confusion and keeps your scripts running clean. Happy coding!

檢查檔案是否存在，別再讓 try-except 把程式碼搞得像迷宮了

來源：Stack Overflow ↗ · 7349 票

還記得剛轉職成後端工程師那年，我負責維護一個處理大批影像的腳本，當時為了檢查某個檔案在不在，我習慣性地寫了滿滿的 try-except 區塊。結果那個腳本一跑起來，邏輯變得超級破碎，當我想除錯時，看著那層層堆疊的例外處理，我簡直快崩潰，根本搞不清楚到底是檔案真的沒找到，還是我的程式碼邏輯在某個角落偷偷拋出了意料之外的錯誤。

其實會遇到這種困擾，是因為 Python 的哲學裡有一句名言叫「請求原諒比請求許可更容易（EAFP）」，這導致很多新手（包括當年的我）會習慣用 try...except FileNotFoundError 來處理檔案檢查。但問題是，當你的業務邏輯變複雜，或者你需要同時檢查幾十個檔案時，這種寫法會讓程式碼的可讀性降到冰點，甚至因為過度濫用例外處理，導致你漏掉了一些真正嚴重的系統錯誤，那種「明明檔案就在那，程式卻一直報錯」的無力感，真的是工程師生涯最痛苦的時刻之一。

後來我學乖了，其實 Python 的標準函式庫已經提供了非常直覺的方法。如果你不想用那些囉唆的 try-except，最推薦的方式就是直接使用 pathlib 模組。這套模組在 Python 3.4 之後就內建了，寫起來非常簡潔優雅。你可以直接這樣寫：


from pathlib import Path

file_path = Path('my_data.txt')

if file_path.is_file():
    print("檔案找到啦，開工！")
else:
    print("檔案不存在，跳過處理。")

如果你是因為維護老專案，還在用 os 模組的話，也可以用 os.path.exists() 來達成一樣的目的，雖然 pathlib 是物件導向的風格更好用，但 os.path 依然是個穩妥的選擇：


import os

if os.path.exists('my_data.txt'):
    # 這裡放你的處理邏輯
    pass

不過，這裡有個經驗談要提醒大家，雖然用這些檢查方法很方便，但千萬要注意「競態條件（Race Condition）」的問題。也就是說，當你檢查完檔案存在後，在下一行程式碼執行前，檔案有可能剛好被刪除或移動了。為了預防這種尷尬的情況再次發生，我通常會建議，如果你的程式邏輯允許，直接嘗試開啟檔案並處理錯誤，或者在進行關鍵操作時加上適當的鎖機制。平常寫程式時，盡量保持邏輯單純，不要過度依賴「檢查」來取代「處理」，這樣你的程式才會長命百歲。

寫程式就是這樣，找到最順手的工具，剩下的時間拿去喝杯咖啡比較實在啦！
