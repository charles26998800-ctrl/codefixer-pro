How to Undo Your Last Local Git Commit Without Losing Your Work

Source: Stack Overflow ↗ · 27226 votes

I still remember the first time I accidentally committed a massive folder of local build artifacts and sensitive config files right before a deadline. I spent about three hours sweating, convinced I’d permanently trashed my branch history, only to realize that Git is actually much more forgiving than it looks when you’re in a panic. It’s a rite of passage for every developer, so if you’re staring at your terminal right now feeling that familiar pit in your stomach, take a deep breath—you’re going to be just fine.

The root cause of this frustration is usually just a simple "fat-finger" moment. When you run git commit, you’re essentially taking a snapshot of your current staging area and locking it into your local repository history. Because Git is designed to be a ledger of your work, it doesn't automatically assume you made a mistake; it treats that commit as a permanent record. If you haven't pushed that commit to a remote server, it’s still entirely contained within your local machine, which is why it’s so easy to fix once you know which command to pull out of your toolkit.

To fix this, I always reach for git reset. The command you want is git reset --soft HEAD~1. This is my go-to because it moves your branch pointer back to the previous commit but keeps all the changes from the commit you just "undid" sitting in your staging area. It’s like hitting a giant undo button that preserves your actual file edits.


# Undo the last commit, but keep your changes staged
git reset --soft HEAD~1

If you realize that you actually made a complete mess of things and you want to wipe those changes entirely—as if that commit never existed and your files should revert to the state they were in before you started editing—you can use the --hard flag instead. Just be careful with this one, because it will permanently discard any uncommitted local changes you were working on.


# WARNING: This destroys your local changes!
git reset --hard HEAD~1

If you’re worried about this happening again, the best habit I’ve picked up is to use git status like a nervous tick. Before I run any commit command, I run git status to see exactly what’s staged. If I see files I didn't mean to include, I use git restore --staged to unstage them before they ever make it into the commit history. It takes an extra five seconds, but it’s saved me from countless headaches over the years.

Hopefully, this gets your repo back in shape so you can get back to writing actual code. Happy coding, and don't sweat the small stuff!

誤提交 Git 怎麼辦？別慌，這招讓你優雅地「時光倒流」

來源：Stack Overflow ↗ · 27226 票

還記得剛進軟體業那年，有次趕著下班，沒檢查就直接 git commit -m "update"，結果一推才發現，我竟然把測試用的假資料庫設定檔給 commit 進去了，更慘的是當時剛好有同事在看這份 repo，那一瞬間我真的心跳漏了一拍，差點以為明天就要打包走人。那種「完蛋了，我把垃圾推上去」的焦慮感，絕對是每個工程師職涯中揮之不去的惡夢，尤其是想到還沒 push 到 server 時，那種不上不下的心情真的超煎熬。

其實這問題的根本原因很單純，就是 Git 的「暫存區（Staging Area）」機制。當你執行 git commit 時，Git 其實是把你的變更打包成一個紀錄節點，但這個節點還留在你的電腦裡。如果你還沒執行 git push，這些錯誤紀錄就像是還沒寄出的信，只要我們知道怎麼把信從郵筒抽回來，一切都還來得及。

解決這個問題最常用的指令就是 git reset。如果你只是想撤銷最後一次 commit，但想保留你辛苦寫好的程式碼檔案（不想讓它們消失），你可以下這個指令：


git reset --soft HEAD~1

這行指令的威力在於，它會把那個 commit 拆掉，但你剛才變更的檔案會全部回到「已暫存（Staged）」的狀態，你只要繼續修改、重新 commit 就可以了。如果你的狀況更慘，連那些檔案都不想要了，想直接把整個 commit 徹底抹掉，讓狀態回到 commit 之前，那就用：


git reset --hard HEAD~1

不過要特別小心，這招會直接把你的修改清空，建議先確認有沒有備份，不然真的會欲哭無淚。

為了不要再經歷這種心臟病發作的時刻，我現在養成了一個習慣：在 commit 之前，絕對會先輸入 git status 看一眼。雖然這看起來很基本，但真的能救命。或者你可以嘗試使用圖形化介面工具，比如 SourceTree 或 GitKraken，在點下 commit 按鈕前，視覺化檢查一下到底改了哪些檔案，這樣就不會再手滑送出錯誤的垃圾代碼了。

Git 的指令雖然看起來冷冰冰的，但只要摸熟了，它其實是你最可靠的後悔藥，下次遇到這種事，先深呼吸別緊張，總有辦法救回來的！
