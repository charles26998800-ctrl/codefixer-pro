How to Fix the "RuntimeError: Input type and weight type should be the same" in PyTorch

Source: Stack Overflow ↗ · 201 votes

I remember the first time I ran into this error back when I was training my first custom CNN. I’d spent an entire afternoon getting my data pipeline ready, finally hit 'run,' and watched my terminal explode with that dreaded RuntimeError about FloatTensor and cuda.FloatTensor mismatch after only two seconds of training. I spent three hours digging through my code, convinced my GPU drivers were fried or my environment was corrupted, only to realize I’d completely forgotten to move my input tensors to the device.

The reason this happens is actually quite straightforward once you stop panicking. PyTorch is incredibly strict about hardware affinity: your model weights and your input data must live on the same device. When you call model.to(device), you’re successfully pushing your neural network onto the GPU, but the data coming out of your DataLoader still defaults to your CPU memory. When the model tries to perform that first matrix multiplication, it finds a mismatch between the data waiting on the CPU and the weights sitting on the GPU, and it throws a hard error to prevent a total memory disaster.

To fix this, you just need to explicitly move your input tensors to the same device as your model during your training loop. I usually handle this right after I unpack my data from the loader. Here is how I set it up in my own projects to make sure everything stays in sync:


device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model.to(device)

for data in dataloader:
    # This is the line that saves the day
    inputs, labels = data
    inputs = inputs.to(device)
    labels = labels.to(device)
    
    outputs = model(inputs)
    # ... rest of your training logic

If you’re working with multi-target tasks, don’t forget to move your labels to the device as well, or you’ll just run into the same error a few lines later when you try to calculate your loss function.

The best way to prevent this from ruining your flow in the future is to write a small helper function or a custom wrapper for your data loading process. I personally like to keep a to_device() function handy that checks if an object has a .to() method before applying it, which keeps my training loops looking clean and prevents me from having to manually call .to(device) on every single tensor I touch. It’s a small bit of boilerplate that’s saved me from countless headaches during late-night debugging sessions.

Hope this gets your training back on track without any more cryptic errors. Good luck with the model training!

PyTorch 報錯：Input type 和 weight type 不一致的崩潰日常

來源：Stack Overflow ↗ · 201 票

還記得剛開始接觸 PyTorch 的時候，那天為了趕一個實驗室的 Demo，熬夜調參調到凌晨三點。好不容易把模型架構寫完，信心滿滿地按下 Run，結果終端機直接噴出一大串紅字，顯示什麼 RuntimeError: Input type (torch.FloatTensor) and weight type (torch.cuda.FloatTensor) should be the same。當下我真的差點把鍵盤砸了，心想：「我不是明明就寫了 .to(device) 了嗎？為什麼還跟我鬧脾氣？」

其實這個錯誤的核心原因非常單純，就是「不同調」。簡單來說，PyTorch 是個很嚴格的強迫症患者，如果你的模型權重（weight）已經被移到了 GPU 上，但你的輸入資料（input）卻還懶洋洋地留在 CPU 的記憶體裡，這兩個傢伙在計算時根本接不上線，PyTorch 就會因為「格式不合」直接罷工。最痛苦的時刻莫過於你在訓練迴圈裡卡了半天，以為是模型結構哪裡寫錯了，結果發現只是因為漏掉了一個小小的 .to(device)，那種懷疑人生的感覺真的會讓人想原地登出。

解決這個問題其實很直觀，你只需要確保資料在餵給模型之前，也跟著一起搬到同樣的裝置上。當你定義好 device 後，記得在訓練迴圈裡對 inputs 和 labels 同樣執行搬移動作。我習慣的寫法是這樣：


device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model.to(device)

for data in dataloader:
    # 關鍵就在這裡！把資料也送到同樣的 device
    inputs, labels = data
    inputs = inputs.to(device)
    labels = labels.to(device)
    
    outputs = model(inputs)
    # 接下來的 loss 計算也要確保 labels 在同一個 device

如果要徹底預防這種惱人的錯誤再次發生，我建議大家養成一個好習慣：不要隨意在程式碼各處硬寫 device。你可以寫一個小小的 helper function，或者在進到訓練迴圈的第一行就統一處理資料搬移。甚至更懶一點的方法，是寫一個簡單的 move_to(data, device) 遞迴函式，這樣不管你的資料是 Tensor、list 還是 dictionary，都可以一次全部推到 GPU 上，再也不用擔心在哪個角落又漏掉一項資料沒搬。
