How to Undo Your Most Recent Local Git Commits Without Losing Your Mind

Source: Stack Overflow ↗ · 27224 votes

I still remember the first time I accidentally committed a massive folder of local build artifacts and sensitive config files right before a demo. I sat there staring at my terminal for a solid ten minutes, sweating bullets because I thought I’d permanently trashed my branch history, and I ended up wasting an entire afternoon trying to manually revert every single file. It’s a rite of passage for every developer, and if you haven’t had that "oh no" moment after hitting enter on a git commit, you probably just haven't been coding long enough.

The reason this happens is usually just a simple lapse in focus or a misunderstanding of Git's staging area. We get into a rhythm, run git add . without checking the status, and fire off a commit before realizing we’ve included files that definitely shouldn't be in version control. Since Git is designed to be a ledger of your history, it doesn't just "forget" that commit unless you explicitly tell it to move the branch pointer back to a previous state. If you haven't pushed your work to a shared repository yet, you're in the perfect position to clean up your mess without anyone else ever knowing you made it.

To fix this, I always reach for git reset. The command you’ll need depends on whether you want to keep your changes or nuke them entirely. If you want to undo the commit but keep your work in your staging area (so you can fix the file selection), just run:


git reset --soft HEAD~1

This moves your branch back one commit but keeps all the files you "committed" in your staging area. If you’ve realized you made a massive mistake and you actually want to discard those changes and revert your files to the state they were in before that commit, use the --hard flag instead:


git reset --hard HEAD~1

Be careful with that hard reset, though—it’s a one-way ticket to deleting your progress. If you’re just looking to unstage files while keeping your commit history intact, you can use git restore --staged to pull specific items out of the staging area before you finalize the commit.

The best way to prevent this headache is to stop using git add . blindly. I’ve started making it a habit to run git status every single time before I commit, just to scan the list of files. It takes three seconds, but it’s saved me from countless hours of cleanup.

Hope this clears up your branch history and saves you some stress today. Happy coding!

手滑 Git commit 怎麼辦？別慌，這招讓你優雅地「時光倒流」

來源：Stack Overflow ↗ · 27224 票

記得有一次，我正趕著要在下班前把功能推上去，結果一時手快，把一堆還在測試階段的垃圾 log 和沒寫完的半成品檔直接 commit 進去了。當時看到終端機跑出那一串 commit 成功訊息，我差點崩潰，心想：「完了，這下要是推到遠端，不就變成全組公審我的半成品了嗎？」那種心臟漏跳一拍、冷汗直流的感覺，真的不是開玩笑的，搞不好還會被同事拿來當成茶餘飯後的笑話。

其實這問題的成因很簡單，就是我們在本地端執行了 git commit，但還沒執行 git push。Git 是一個很強大的版本控制工具，它允許我們在本地「反悔」。當你 commit 下去的那一刻，其實只是把變更記錄在了本地的 git 物件庫裡，只要還沒推到遠端伺服器，這些記錄就完全在你的掌控範圍內，想怎麼改都行。

當我發現自己 commit 錯東西時，我通常會用 git reset 來處理。如果你只是想撤銷最後一次 commit，但想把那些改動留下來繼續調整，我會用 git reset --soft HEAD~1。這個指令會把 commit 記錄刪掉，但你剛剛寫好的程式碼會完好無損地留在 staging area，就像是你剛打完程式碼、還沒 commit 的狀態，這樣我就能補上忘記加入的檔案，或是把不該加的檔案移除，然後重新 commit 一次。

如果你的情況更糟，連程式碼都寫爛了，想徹底毀屍滅跡、回到 commit 前的乾淨狀態，那就用 git reset --hard HEAD~1。但這招真的要小心，它會把你 commit 之後的所有修改全部「原地消失」，那種按下 Enter 後發現程式碼沒救回來的痛苦，我可是領教過的。所以除非你確定那些改動都是垃圾，否則千萬別亂用 --hard。

為了避免這種心驚膽跳的時刻再次發生，我現在養成了習慣，每次 commit 前一定會先打 git status 看一眼。看清楚現在到底有哪些檔案在「舞台上（Staging Area）」，甚至養成使用 git diff --cached 的好習慣，檢查一下這次 commit 到底包含哪些變更，才不會下次又發生這種「手滑悲劇」。

學會這幾招，以後遇到這種狀況就不用再對著螢幕乾瞪眼啦，趕快去試試看吧！
