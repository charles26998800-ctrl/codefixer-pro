How to Use Local Docker Images with Minikube Without the Registry Headache

Source: Stack Overflow ↗ · 688 votes

I remember the first time I set up Minikube, I spent a solid three hours banging my head against the wall because my pods were stuck in ImagePullBackOff. I was convinced my YAML was broken, but it turned out I was just pushing images to my local Docker daemon and expecting Minikube to magically see them inside its own isolated VM. It’s one of those "aha!" moments every Kubernetes dev hits, and honestly, it’s frustrating as hell when you just want to iterate quickly on your code.

The reason this happens is simple: Minikube runs inside its own virtual machine or container. When you run docker build on your laptop, that image lives in your host's Docker daemon. Minikube has zero access to that daemon; it’s looking at its own internal registry, which is currently empty. It tries to pull the image from Docker Hub, fails because it doesn't exist there, and you’re left staring at a crash loop.

The fastest way I found to fix this is to point your local terminal directly at the Minikube Docker daemon. Instead of trying to push images to a registry, you just tell your shell to "talk" to Minikube's internal Docker environment. You can do this by running this command in your terminal:


eval $(minikube -p minikube docker-env)

Once you run that, any docker build or docker tag command you execute will happen directly inside the Minikube environment. Now, when you run your kubectl apply command, the cluster sees the image locally and starts the container instantly. Just remember that if you open a new terminal tab, you’ll need to run that eval command again, or your terminal will revert to your host's Docker daemon.

To keep this from biting you again, the best habit I’ve picked up is to keep my image imagePullPolicy set to IfNotPresent in my Kubernetes manifests. By default, K8s often tries to pull the latest version of an image, which forces a network request even if you have the image locally. Setting it to IfNotPresent tells the node, "Hey, if you’ve got it, use it," which saves you from those annoying registry timeouts during local development.

Hope this saves you a few hours of debugging time today. Happy coding!

別再把 Image 推到 Docker Hub 了！Minikube 使用本地映像檔的正確姿勢

來源：Stack Overflow ↗ · 688 票

記得剛開始碰 Kubernetes 的時候，為了測試一個小小的 API 修改，我竟然天真地把 Docker Image 推到 Docker Hub，再讓 Minikube 去 pull 下來。結果那個下午我都在跟網路速度搏鬥，看著終端機顯示 ImagePullBackOff 的那一刻，我真的差點要把鍵盤砸了，心想：「我只是改了兩行程式碼，難道要花十分鐘等部署嗎？」

會發生這種事，是因為 Minikube 其實是一個跑在虛擬機（或容器）裡的獨立環境。當你在電腦上執行 docker build 時，映像檔是存在你主機的 Docker Daemon 裡，但 Minikube 的那台虛擬機根本看不見你的主機。這就像是你把便當放在自己家冰箱，卻指望樓下鄰居能直接拿去吃一樣，完全是兩個獨立的世界，難怪 Kubernetes 找不到東西。

後來我發現，其實根本不需要繞遠路去推送到遠端倉庫。最快的方法就是直接告訴 Minikube：「欸，用我現在這個 Docker 環境！」你只要在執行 build 指令前，先下這行指令切換環境變數：


eval $(minikube docker-env)

這行指令跑完之後，你再執行 docker build -t my-app:v1 .，這個映像檔就會直接被建立在 Minikube 內部的 Docker 環境裡了。這時候你只要把 Deployment 裡的 imagePullPolicy 設定成 IfNotPresent，Minikube 就會直接拿本地現有的檔案來跑，部署速度快到讓你懷疑人生，完全不需要再等那漫長的下載過程。

為了避免以後忘記，或者哪天又鬼打牆跑去 push 遠端，我現在習慣在專案的 Makefile 裡直接寫好這一套流程。我建議你也養成習慣，在開發階段就把這種指令整合進自動化腳本裡，甚至可以在 imagePullPolicy 這塊多留意一下，確保它不會在每次重啟時都嘗試去網路上找新版本。

現在我改完程式碼只要一秒鐘就能看到 Pod 重啟，再也不用在那邊盯著 ImagePullBackOff 發呆了，這感覺真的爽太多。
