How to Fix the "Input type and weight type should be the same" Error in PyTorch

Source: Stack Overflow ↗ · 201 votes

I remember the first time I ran into this error; I’d just finished setting up my first multi-GPU training script and felt like a total genius. I hit run, watched the progress bar for half a second, and then my terminal exploded with that dreaded RuntimeError. I spent three hours refactoring my model architecture and checking my dataset paths, only to realize I’d completely forgotten to move the actual data to the GPU. It’s one of those "face-palm" moments that every deep learning engineer goes through at least once.

Basically, this error happens because PyTorch is strictly enforcing device compatibility. When you call model.to(device), you’re successfully moving your model weights into your GPU’s VRAM, but your input data—the tensors coming out of your DataLoader—are still sitting comfortably in your CPU's RAM by default. When the math happens in the first layer, PyTorch tries to perform an operation between a CPU tensor and a GPU tensor, and it throws an error because those two things physically can’t talk to each other in that context. It’s like trying to plug a European power adapter into a US wall outlet; the connection just isn't happening.

The fix is straightforward, though it’s easy to overlook. You need to explicitly push your data tensors to the same device your model is using inside your training loop. I usually handle this right after I unpack my data from the loader. Here’s the snippet that saves the day:


device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model.to(device)

for inputs, labels in dataloader:
    # Move the data to the same device as the model
    inputs = inputs.to(device)
    labels = labels.to(device)
    
    outputs = model(inputs)
    # ... rest of your training logic

Once you add those .to(device) calls, your data and your weights will be living on the same hardware, and the error will vanish instantly.

To keep this from biting you again, I’ve made it a habit to define a helper function or a simple wrapper for my training steps that automatically handles device assignment. If you get into the rhythm of moving your inputs to the device immediately after they leave the DataLoader, you won't ever have to worry about this mismatch again.

Hope this clears up your terminal logs so you can get back to training! Let me know if you run into any other weird CUDA quirks.

救命！PyTorch 的 GPU 裝置不匹配地獄：RuntimeError: Input type and weight type should be the same

來源：Stack Overflow ↗ · 201 票

還記得剛開始接觸 PyTorch 的時候，那天我興沖沖地寫好了一個複雜的模型，想說終於可以丟到 GPU 上跑出驚人的訓練速度了。結果按下執行鍵後，看著終端機噴出那行大大的 RuntimeError: Input type (torch.FloatTensor) and weight type (torch.cuda.FloatTensor) should be the same，我整個人直接原地崩潰，心想：「明明都已經寫了 .to(device) 啊，到底是哪裡冒出來的叛徒？」

其實這問題的根本原因說穿了很單純，就是你的模型雖然已經成功「移民」到 GPU 了，但你餵給它的資料（也就是 inputs）卻還留在 CPU 的舒適圈裡不想動。PyTorch 是個很嚴格的強迫症患者，它要求參與運算的每一個 Tensor 都必須在同一個「物理空間」——也就是同一個裝置上。如果你拿著 CPU 的資料去撞 GPU 的參數，它當然會毫不留情地給你報錯。最讓人痛苦的是，這種錯誤通常發生在訓練迴圈的中間，你看著模型跑了前面幾步以為穩了，結果在某個資料批次突然崩掉，那種浪費掉的時間和算力真的會讓人想把螢幕砸了。

要解決這個問題，其實只要在迴圈裡把資料也「送過去」就好了。我當時修正的方式很直接，就在取得資料後，強制把 inputs 和 labels 都丟到我定義好的 device 上。你可以直接參考這段寫法：


device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model.to(device)

for data in dataloader:
    # 關鍵就在這裡！要把資料也丟到跟模型同一個裝置
    inputs, labels = data
    inputs = inputs.to(device)
    labels = labels.to(device)
    
    outputs = model(inputs)
    # 接下來的 loss 計算也記得要在同一個裝置上處理喔

至於要怎麼預防下次再踩進同樣的坑？我現在養成了一個習慣，就是寫一個簡單的輔助函式，或者是直接在定義 Dataset 時就把 to(device) 的邏輯封裝好。但最簡單的做法，還是把「將輸入資料移動到裝置」這件事，當作寫訓練迴圈時的第一行反射動作，寫久了就像吃飯呼吸一樣自然，根本不用大腦思考。
