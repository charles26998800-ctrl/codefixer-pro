The Pythonic Way: Checking if a File Exists Without Try-Except Blocks

Source: Stack Overflow ↗ · 7349 votes

I remember back when I was a junior dev working on my first file-parsing script, I was obsessed with wrapping every single operation in a try-except block. I spent nearly three hours one afternoon debugging a simple log-scraper because I’d nested so many handlers that the actual error was buried under five layers of generic exceptions. I felt like a total amateur when I realized I was over-engineering a check that should have been a simple boolean gate.

The reason we often reach for try-except for file checks is that it's the "Pythonic" way to handle errors—the EAFP (Easier to Ask for Forgiveness than Permission) principle. However, when you’re just trying to verify the existence of a file to decide whether to trigger a process or skip it, EAFP can feel clunky and verbose. The root cause of the confusion is usually that we want to keep our control flow clean, and having a FileNotFoundError bubble up just to tell us a file is missing feels like overkill when you could just ask the filesystem directly before you even touch the file.

When I need to check for a file without the overhead of exception handling, I reach for the pathlib module. It’s been the standard since Python 3.4, and it makes the code incredibly readable. You don't need to import os or mess around with os.path.exists(), which feels a bit dated at this point. Here is how I handle it in my current projects:


from pathlib import Path

file_path = Path("config.json")

if file_path.is_file():
    print("The file exists, let's load it.")
else:
    print("File not found, skipping.")

If you’re working with an older codebase or just prefer the legacy style, the os module is still perfectly valid. I still see it in plenty of production systems I maintain. You’d do it like this:


import os

if os.path.exists("config.json"):
    # Do your work here
    pass

The best way to prevent these issues from spiraling into a mess is to decouple your "check" from your "action." Even if you verify the file exists, a race condition can happen where the file is deleted or locked by another process a millisecond later. My tip is to use is_file() to control your application flow, but always keep a try-except block around the actual open() call. It’s the only way to be 100% safe, no matter how much you check beforehand.

Hope this clears up the confusion and helps you write some cleaner code today. Happy coding!

別再為了檢查檔案存不存在寫滿滿的 try-except 了！

來源：Stack Overflow ↗ · 7349 票

記得剛入行那年，我寫了一個自動化搬移檔案的腳本，當時為了確保程式不會因為讀不到設定檔就直接噴錯誤掛掉，我寫了一堆 try...except FileNotFoundError 的邏輯。結果整個程式碼看起來簡直像是一團亂碼，最慘的是，有次因為環境變數設錯，我誤以為檔案存在，結果程式在 try 區塊裡卡住，除錯時看著那幾百行錯誤訊息，我真的差點在辦公室崩潰，完全找不到到底是哪個環節沒對上。

其實這問題的根本原因，在於 Python 對於「異常處理」的哲學跟其他語言不太一樣。很多人習慣「先做再說，出錯了再抓」，但如果你只是想確認檔案在不在，還要寫完整的錯誤捕捉邏輯，確實會讓程式碼變得很髒，而且如果你的 try 區塊包得太大，很容易不小心把其他重要的錯誤也一起吞掉，導致你根本不知道程式其實是在別的地方壞掉的。

後來我學乖了，直接改用 pathlib 模組，這根本是現代 Python 開發者的救星。現在我檢查檔案時，都會直接用 path.exists()，它不僅語意明確，讀起來就像在講英文一樣直覺。如果你還在用舊版的 os.path，真心建議你轉換陣營，程式碼會乾淨非常多。寫法大概是長這樣：


from pathlib import Path

file_path = Path("config.json")

if file_path.exists():
    print("檔案還在，可以放心讀取！")
else:
    print("檔案不見了，趕快去檢查路徑。")

如果要確保它不只存在，還必須是一個「檔案」（而不是一個剛好同名的資料夾），我會多加一個 is_file() 的判斷，這樣就萬無一失了。

為了避免以後再發生這種「明明檢查了卻還是報錯」的狀況，我現在養成一個習慣：絕對不在檢查檔案跟讀取檔案之間做太多複雜的運算。因為就算你檢查時檔案在，下一毫秒它也有可能被其他程序刪除（這叫 Race Condition），所以如果你真的要進行寫入或高頻率的讀取操作，該處理的錯誤還是要處理，但對於簡單的邏輯判斷，用 pathlib 就已經綽綽有餘了。

希望這些小技巧能幫你的程式碼瘦身成功，別再被那些冗長的例外捕捉給綁架啦！
