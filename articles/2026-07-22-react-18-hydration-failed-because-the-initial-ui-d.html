How to Fix the "Hydration Failed" Error in Next.js and React 18

Source: Stack Overflow ↗ · 341 votes

I still remember the first time I saw the dreaded "Hydration failed" error in a production build. I’d spent three hours debugging what I thought was a complex state management issue, only to realize I had a simple Date() call in my JSX that was outputting slightly different timestamps on the server versus the client. I wanted to throw my keyboard out the window when I realized my entire app was breaking because of a single millisecond difference.

The root cause of this frustration is actually quite logical once you look past the scary console error. When you use Next.js, your server pre-renders your HTML and sends it to the browser. React then takes over on the client side, "hydrating" that static HTML by attaching event listeners and making it interactive. The problem happens when the HTML generated by the server doesn't perfectly match the HTML that React renders on the client during that first pass. If you're using something like typeof window checks, random number generators, or browser-specific APIs (like localStorage or window.innerWidth) directly in your render logic, the server and client will disagree on what the UI should look like, and React just throws its hands up and gives up.

To fix this, I’ve learned that you have to delay the execution of client-specific code until the component has actually mounted. I usually reach for a simple useEffect hook to toggle a "mounted" state. By checking if the component is mounted before rendering the problematic UI, I ensure that the server-side render stays consistent (usually by rendering null or a loading state) and the client-side render takes over only after the hydrate phase finishes.


import { useState, useEffect } from 'react';

export default function MyComponent() {
  const [isMounted, setIsMounted] = useState(false);

  useEffect(() => {
    setIsMounted(true);
  }, []);

  if (!isMounted) {
    return null; // Or a simple skeleton loader
  }

  return (
    <div>
      {/* Now you can safely use window or browser-only APIs */}
      Current window width: {window.innerWidth}
    </div>
  );
}

inside a

tag or placing a

in a way that causes the browser to auto-correct the DOM structure. Since the server sends the original code but the browser "fixes" it during parsing, the DOM tree mismatch triggers the exact same hydration warning. Always validate your markup against the HTML spec to ensure the browser isn't silently changing your code before React gets a chance to look at it.

The best way to prevent this from ruining your day is to keep your UI components "pure" whenever possible. If you need to use browser-only features, try to isolate them into specific client-side components and keep your server-rendered shell as clean as possible. It makes your code a lot easier to test and saves you from those late-night debugging sessions.

Hope this saves you a few hours of head-scratching. Happy coding!

救命！Next.js 報錯 Hydration failed：當伺服器跟瀏覽器開始吵架

來源：Stack Overflow ↗ · 341 票

記得那天凌晨兩點，我正趕著要把一個 Side Project 上線，結果刷新頁面後，控制台直接炸出一整排紅通通的「Hydration failed」錯誤，當時真的差點沒把鍵盤給砸了。明明程式碼看起來沒問題，伺服器渲染也正常，但瀏覽器就是死活不買單，那種「我明明寫對了，你憑什麼說我不對」的無力感，相信每個寫過 React 的人一定都懂。

其實這問題說穿了，就是伺服器（Server）跟瀏覽器（Client）在鬧彆扭。當你用 Next.js 這種 SSR 框架時，伺服器會先跑一次程式碼，生成一份 HTML 給瀏覽器；接著瀏覽器收到後，會再跑一次同樣的程式碼來「接管」頁面，也就是所謂的 Hydration。如果這兩邊吐出來的內容有一丁點對不上，比如你在伺服器端用了 new Date() 或是隨機數字，瀏覽器跑出來的結果跟伺服器給的不一樣，React 就會崩潰，直接放棄治療跳出報錯。

我當時遇到的情況，其實是因為我在元件裡直接用了 window 物件，或者是一些依賴當下時間的邏輯。伺服器渲染時根本沒有 window，所以它直接略過或給了預設值，結果瀏覽器一跑，發現兩邊的 DOM 結構完全對不起來。解決方法其實不難，最暴力但也最有效的方式，就是確保這個元件「只在客戶端渲染」。我習慣的做法是利用 useEffect 來處理，因為這段程式碼保證只會在瀏覽器環境執行，伺服器根本碰不到。


import { useState, useEffect } from 'react';

const MyComponent = () => {
  const [isClient, setIsClient] = useState(false);

  useEffect(() => {
    setIsClient(true);
  }, []);

  if (!isClient) {
    return null; // 或者你可以放一個 Loading 的轉圈圈
  }

  return <div>這裡現在只有瀏覽器才看得到，再也不會吵架了！</div>;
};

為了避免以後再被這個 Bug 搞到崩潰，我現在養成了「防禦性編碼」的習慣。只要是會用到 localStorage、window 這種只有瀏覽器才有的東西，我一定會先在元件最外層包一層「檢查機制」。或者乾脆在 Next.js 的元件引入時，使用 dynamic 並設定 ssr: false，直接告訴 Next.js：「這傢伙別在伺服器跑了，省得麻煩」。

希望這些經驗能幫你省下幾個小時的除錯時間，趕快去把你的專案修好吧，別再讓伺服器跟瀏覽器吵架了！
