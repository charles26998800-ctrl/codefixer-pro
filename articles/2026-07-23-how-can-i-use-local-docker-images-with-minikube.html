How to Use Local Docker Images with Minikube Without the Registry Headache

Source: Stack Overflow ↗ · 688 votes

I still remember the first time I set up Minikube, feeling pretty proud of myself for containerizing my microservice. I spent three hours banging my head against the wall because my pod kept stuck in ImagePullBackOff, even though I was staring right at the image in my local Docker daemon. It was infuriating—I felt like I was taking crazy pills until I realized that Minikube lives in its own little world, completely oblivious to what’s happening on my host machine’s Docker engine.

The reason this happens is actually quite logical once you peek under the hood. Minikube runs inside a virtual machine or a containerized environment of its own. When you build an image on your laptop, that image is stored in your host's Docker daemon. Minikube, by default, tries to pull images from a remote registry (like Docker Hub) because it doesn't have a direct line to your local cache. It’s essentially a security and isolation boundary; your Kubernetes cluster doesn't automatically trust or see the files on your host OS.

The cleanest way I’ve found to bridge this gap is by pointing your local terminal directly at the Docker daemon running inside Minikube. Instead of pushing to a registry and pulling back down, you tell your shell to run commands against the Minikube environment. You can do this by running the following command in your terminal:


eval $(minikube docker-env)

Once you run that, any docker build command you execute will build the image directly inside the Minikube node. You can verify it worked by running docker images, and you'll see your custom build sitting right there waiting for Kubernetes. From that point on, you just need to ensure your Kubernetes deployment manifest has imagePullPolicy: IfNotPresent set. This tells Kubernetes to check the local node cache first, see that the image is already there, and skip the attempt to reach out to the internet.

To keep your sanity, I recommend getting into the habit of using a specific naming convention or tags for your local development images. If you’re constantly iterating, it's easy to get confused between a local build and a remote one. If you keep your local images tagged as my-app:dev, you’ll always know exactly what you’re deploying without wondering if you accidentally pulled an old version from Docker Hub.

Hope this saves you from the same three-hour headache I had. Happy coding, and may your pods always reach the Running state on the first try!

別再傻傻推送到 Docker Hub 了！在 Minikube 使用本地映像檔的正確姿勢

來源：Stack Overflow ↗ · 688 票

記得剛開始接觸 Kubernetes 時，我為了測試一個小小的 API 改動，每次都要把 Docker image build 好，然後 docker push 到 Docker Hub，再等 Minikube 把那個肥大的 image 下載回來。那時候網路又慢，光是等這幾分鐘的循環，我咖啡都喝掉三杯了，最後看到 Pod 還是 ImagePullBackOff 的那一刻，我真的差點把鍵盤砸了，那種「我明明就在本地改好程式了，為什麼你就是讀不到！」的崩潰感，相信你也懂。

其實這背後的根本原因很簡單：Minikube 是一個跑在虛擬機（或是容器）裡的獨立環境，它的 Docker Daemon 跟我們宿主機（Host）的 Docker Daemon 是完全隔離的。當我們在宿主機下 docker build 時，鏡像只是存在你的電腦裡，Minikube 裡面的節點根本「看不到」這個圖書館。它預設會嘗試去 Docker Hub 抓取，找不到當然就報錯，這就是為什麼你怎麼改程式，Minikube 永遠都用舊版本的原因。

要解決這個問題，其實根本不需要什麼複雜的設定，最快的方法就是讓你的終端機直接「對準」Minikube 內部的 Docker 環境。只要在終端機輸入這行指令，讓環境變數指向 Minikube 的 Docker Daemon，接下來你 build 出來的 image 就會直接進到 Minikube 的肚子裡：


eval $(minikube docker-env)

執行完這行後，你再下 docker build -t my-app:v1 .，就會發現 build 速度飛快，因為它直接存進了 Minikube 的映像檔列表。這時候記得把 Deployment YAML 裡的 imagePullPolicy 改成 IfNotPresent，不然 Kubernetes 還是會雞婆地跑去檢查網路上的版本。當然，如果你不想每次開終端機都打這行，也可以把它加到 .bashrc 或 .zshrc 裡，但要記得，這只對當前視窗有效。

為了避免這種「我明明 build 了卻跑不出來」的痛苦，我後來養成了一個習慣：只要是在本地開發，我一定會先把 imagePullPolicy 設為 IfNotPresent。這樣就算哪天網路斷了，或者 Minikube 突然發神經，它也只會優先使用本地已經有的版本，不會沒事跑去外面找麻煩。

搞定這個之後，開發速度真的快到飛起，再也不用看著進度條發呆了，祝大家除錯順利！
