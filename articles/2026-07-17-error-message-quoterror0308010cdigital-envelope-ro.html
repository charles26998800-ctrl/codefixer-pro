How to Fix the "digital envelope routines::unsupported" Error in React

Source: Stack Overflow ↗ · 2573 votes

I still remember the day I upgraded my local development environment to Node.js v17. I was mid-sprint, ready to knock out a simple feature for a client project, when I ran npm start and was greeted by a wall of red text screaming about "digital envelope routines." I spent a good three hours convinced I’d somehow corrupted my entire node_modules folder or that my OS had finally decided to retire, only to realize it was just a massive breaking change I hadn't prepared for.

If you’re seeing this, it’s because Node.js v17 and later versions integrated OpenSSL 3.0, which tightened up security protocols significantly. Many older libraries—and even some legacy React build configurations—rely on outdated algorithms like MD4, which OpenSSL 3.0 now flags as insecure and "unsupported" by default. Essentially, your project is trying to use a cryptographic hash method that Node no longer trusts, so it throws a tantrum and kills the build process entirely.

The quickest way I’ve found to squash this bug is to force Node to be a bit more lenient by using the legacy OpenSSL provider. You don't need to downgrade your entire Node installation; you just need to set an environment variable. If you’re on a Mac or Linux machine, you can run this right in your terminal before starting your dev server:


export NODE_OPTIONS=--openssl-legacy-provider
npm start

If you’re working on Windows, the syntax is a little different depending on whether you’re using Command Prompt or PowerShell. For PowerShell, you’ll want to run this command:


$env:NODE_OPTIONS = "--openssl-legacy-provider"
npm start

For a more permanent fix so you don't have to type that every single time you sit down to code, I usually just update my package.json file. You can inject that variable directly into your start script so it's baked into your workflow:


"scripts": {
  "start": "react-scripts --openssl-legacy-provider start",
  "build": "react-scripts --openssl-legacy-provider build"
}

To keep this from biting you again, my best advice is to keep your dependencies updated. This error is almost always a sign that your build tools (like react-scripts or your bundler) are lagging behind the version of Node you have installed. Checking npm outdated every once in a while and bumping those packages can save you from these "ghost in the machine" moments.

Hope this saves you the headache it caused me, and happy coding!

遇到 error:0308010C 別慌，這只是 Node.js 版本跟舊專案鬧脾氣而已

來源：Stack Overflow ↗ · 2573 票

那天我剛換了一台新電腦，想說把手邊的一個舊專案拿出來跑跑看，結果一執行 npm start，終端機直接跳出一大串紅色錯誤訊息，那個 digital envelope routines::unsupported 簡直像是在嘲笑我。我當下心裡真的是「髒話消音」，明明昨天在舊電腦上還跑得好好的，怎麼換個環境就直接原地崩潰，當下真的覺得自己這幾年的開發經驗是不是都白費了，連個專案都起不來。

其實這個問題是因為 Node.js 在 v17 版本之後，把 OpenSSL 的版本升級到了 3.0，而這個新版本對一些舊的加密演算法管得比較嚴，偏偏我們很多舊的 React 專案（特別是還在用 Webpack 4 的那種）底層用的編譯工具還在用舊的那套，兩邊直接「雞同鴨講」。簡單來說，就是 Node.js 覺得你的舊專案在用一些「過時的加密方式」，所以為了安全直接幫你擋下來，導致編譯過程卡死。

當下我發現這個問題後，最快的解決方式就是告訴 Node.js：「嘿，請暫時對這些舊程式寬容一點」。你只要在執行指令前加上一個環境變數設定，讓 Node.js 啟用舊版的 OpenSSL 兼容模式就行了。如果你是在 Mac 或 Linux 上，直接在終端機輸入這行：


export NODE_OPTIONS=--openssl-legacy-provider

如果你是用 Windows 的 PowerShell，指令則是：


$env:NODE_OPTIONS = "--openssl-legacy-provider"

設定完之後，再試著跑一次 npm start，通常就能順利看到熟悉的編譯進度條在跑了。為了不用每次打開終端機都要重新打一次這串指令，我建議直接把這段加到 package.json 的 scripts 區塊裡，把 start 改成這樣：


"scripts": {
  "start": "react-scripts --openssl-legacy-provider start",
  "build": "react-scripts --openssl-legacy-provider build"
}

至於要怎麼預防下次再踩到這個坑？說真的，最好的辦法就是定期去更新專案的依賴套件。很多時候這種錯誤都是因為 react-scripts 或 webpack 太舊了，如果專案允許的話，試著升級到較新的版本，讓它們能支援 OpenSSL 3.0，這樣就不需要動不動就開這個「後門」了。當然，如果你是維護那種好幾年沒動的「考古專案」，那這招環境變數設定絕對是你最好的好朋友。

好啦，解決完這個惱人的紅字，趕快回去把剩下的功能寫完，別讓一個 Bug 耽誤了下班時間！
