How to Fix the "Input type and weight type should be the same" Error in PyTorch

Source: Stack Overflow ↗ · 201 votes

I still remember the first time I saw this RuntimeError while training a custom CNN. I had just moved my training loop over to a shiny new GPU instance, feeling like a genius, but as soon as I hit run, my terminal exploded with that cryptic "Input type (torch.FloatTensor) and weight type (torch.cuda.FloatTensor) should be the same" message. I spent three hours staring at my GPU allocation code, convinced my drivers were broken or my PyTorch installation was corrupted, only to realize I’d completely forgotten to move the data itself to the device.

The root cause here is actually pretty simple once you stop pulling your hair out. PyTorch is incredibly strict about where your tensors live. When you call model.to(device), you are successfully moving all the internal weights and biases of your neural network onto your GPU. However, the tensors coming out of your DataLoader are still sitting in your system's RAM (the CPU). When you pass those CPU-bound tensors into a model expecting GPU-bound inputs, PyTorch throws an error because it can't perform math operations across different memory spaces. It’s essentially the software equivalent of trying to plug a European power adapter into an American wall outlet—the pieces just don't fit together.

To fix this, you need to explicitly move your input tensors to the same device as your model during your training or inference loop. It’s an easy one-liner to add, but it’s the most common "gotcha" for anyone moving from CPU to GPU development. Here is how I usually handle it in my training loops:


device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model.to(device)

for data in dataloader:
    # This is the line you're missing!
    inputs, labels = data
    inputs = inputs.to(device)
    labels = labels.to(device)
    
    outputs = model(inputs)
    # ... rest of your training logic

If you’re working with a multi-output model or a more complex data structure, just make sure every tensor that interacts with the model is sent to the device. I’ve found that the cleanest way to prevent this from happening in the future is to write a small helper function or a custom Dataset collate function that handles device placement automatically. If you make it a habit to check the .device attribute of your tensors before they hit the model, you'll save yourself a lot of late-night debugging sessions.

Hope this saves you a few hours of head-scratching today. Happy coding!

PyTorch 報錯：Input type 和 weight type 不一致的解法與防坑指南

來源：Stack Overflow ↗ · 201 票

那天晚上我正盯著螢幕，好不容易把模型架構寫好，滿心期待按下執行鍵，想說總算可以去泡杯咖啡休息了，結果才跑不到一秒，終端機就噴出一大串紅通通的 RuntimeError。看著那個「Input type (torch.FloatTensor) and weight type (torch.cuda.FloatTensor) should be the same」的錯誤訊息，我當下真的是差點在辦公室崩潰，明明已經把模型丟進 GPU 了，為什麼 PyTorch 還是跟我說它們「不對盤」？

其實這個問題的核心概念很簡單，就是 PyTorch 非常嚴格，它不允許你的模型參數在 GPU 上，但你的資料卻還在 CPU 裡面「原地踏步」。當你呼叫 model.to(device) 時，你只是把模型參數搬到了顯示卡記憶體，但你的 dataloader 讀取出來的資料，預設還是會待在 CPU 的系統記憶體裡。就像是你把大廚送進了廚房（GPU），但卻沒把食材（inputs）送過去，廚師沒辦法做菜，當然就直接罷工給你看。

解決方法其實沒什麼魔法，重點就是要把每一筆從 dataloader 撈出來的資料，也同步用 .to(device) 指令搬過去。我當初修改的方式很單純，就是在迴圈裡多加一行：


device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model.to(device)

for data in dataloader:
    # 這裡就是關鍵：把資料也搬到跟模型一樣的地方
    inputs, labels = data
    inputs = inputs.to(device)
    labels = labels.to(device)
    
    outputs = model(inputs)
    # 接下來就可以繼續算 loss 之類的了

每次看到這種錯誤，我都覺得自己好像在玩「傳送門」遊戲，模型在顯卡那頭，資料在 CPU 這頭，兩邊沒連起來就是動不了。最痛苦的時刻通常是模型訓練到一半才報錯，或者是寫到一半忘記把 labels 也搬過去，結果 loss.backward() 的時候又噴出類似的型別錯誤，那種要改不改的感覺真的會讓人很想把鍵盤砸了。

為了避免以後再被這個低級錯誤卡住，我現在習慣寫一個簡單的 move_to_device 函式，或者直接在資料讀取的地方統一處理。更懶一點的做法是寫一個小工具函式，把所有變數自動轉移，這樣寫 code 的時候就不用在那邊一直重複貼 inputs.to(device)，真的能省下不少除錯的力氣。

希望這篇筆記能幫你少走點冤枉路，趕快去把你的 GPU 跑滿吧！
